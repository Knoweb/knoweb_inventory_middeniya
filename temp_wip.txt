Architecture & Design Document: KNOWEB ERP Manufacturing Extension 1. Executive Summary This document outlines the architectural modifications and new service designs required to extend the existing KNOWEB Inventory Management System into a manufacturing-capable ERP. The system will support multi-stage production workflows, dynamic invoicing, supplier mapping, and strict role-based access control. 2. High-Level Architecture The system will utilize a  Microservices Architecture  built on Spring Boot, communicating asynchronously via an Event Broker (e.g., Apache Kafka or RabbitMQ) to handle manufacturing workflows. The frontend will remain a Single Page Application (SPA) using React, communicating with the backend via an API Gateway. Core Components API Gateway (Spring Cloud Gateway):  Handles routing, cross-cutting concerns (CORS, Rate Limiting), and token validation. Identity & Access Management (IAM):  Issues JWTs containing specific role claims (ROLE_STOCK_KEEPER, ROLE_INJECTION_MOLDING, etc.).   Event Broker:  Facilitates decoupled communication between the Order, Inventory, and Notification domains. Frontend (React):  Implements dynamic, role-based routing to serve dedicated dashboards. 3. Microservices Breakdown & Responsibilities To maintain high cohesion and low coupling, the monolithic inventory system should be logically divided (or physically split) into the following domains: A. Catalog & Supplier Service Responsibilities:  Manages the master list of products, auto-generates Item  Codes ,  and handles Supplier profiles. Key Feature:  Maintains the "Supplier-Item Mapping" to define exactly which raw materials are purchased from which vendor. B. Procurement & Inventory Service Responsibilities:  Manages Purchase Orders (POs), Stock In events, and overall stock valuation. Key Feature:  Allows multi-item PO creation with individual line-item pricing. Segregates damaged vs. undamaged items. C. Manufacturing Workflow Service (New) Responsibilities:  Tracks Work  In  Progress (WIP) across the production pipeline (Injection Molding -> Assemble -> Primary). Key Feature:  Manages state transitions of batches. D. Order & Invoicing Service Responsibilities:  Tracks Sales Orders and Purchase Orders through states (Processing, Pending, Complete). Key Feature:  Generates Tax Invoices (inclusive of VAT, SSC) for registered entities and Normal Invoices for unregistered customers, with optional discount calculations. Manages Delivery orders, logging delivery dates, and handling proof-of-delivery photos. E. HR & Finance Service (New) Responsibilities:  A lightweight service strictly for managing and tracking employee advance payments. 4. Event-Driven Workflow (EDA) To prevent tight coupling between the warehouse and the manufacturing floor, the system will rely on domain events. Event Name Publisher Subscriber Action Triggered StockInRecorded Procurement Service Workflow Service, Analytics Updates global stock levels and recalculates financial valuation. WipBatchCompleted Workflow Service Notification Service Alerts the next department in the chain (e.g., Molding alerts Assembly) that a batch is ready. OrderDelivered Order Service Invoicing Service Triggers the auto-generation of the appropriate Tax or Normal Invoice based on customer profile. 5. Core API Specifications (RESTful) Below is the foundational API contract for the new modifications. All endpoints require a valid JWT with appropriate role claims. Supplier & Procurement APIs Method Endpoint Description Required Role POST / api /v1/suppliers Creates a new supplier profile. ADMIN, PURCHASING POST / api /v1/suppliers/{id}/mappings Maps specific item codes to a supplier. ADMIN, PURCHASING POST / api /v1/purchase-orders Creates a PO. Payload includes List< LineItem > with specific prices. PURCHASING Inventory & Workflow APIs Method Endpoint Description Required Role POST / api /v1/inventory/stock-in Records incoming stock and triggers  StockInRecorded  event. STOCK_KEEPER  GET / api /v1/inventory/ wip Fetches half-finished goods currently in WIP state. MOLDING, ASSEMBLE PUT / api /v1/workflow/batches/{id}/advance Moves a WIP batch to the next production stage. MOLDING, ASSEMBLE GET / api /v1/inventory/valuation Returns real-time financial value of stock. ADMIN, FINANCE Order & Invoicing APIs Method Endpoint Description Required Role PATCH / api /v1/orders/{id}/status Updates order state (Processing, Pending, Complete). SALES, ADMIN GET / api /v1/orders/pending/summary Generates a summary of all pending orders. SALES, ADMIN POST / api /v1/deliveries/{ orderId } Logs delivery. Payload includes  deliveryDate  and  MultipartFile  for photos. LOGISTICS POST / api /v1/invoices/generate/{ orderId } Generates invoice. Reads customer tax config to output Tax/Normal type. FINANCE, SALES HR APIs Method Endpoint Description Required Role POST / api /v1/ hr /advances Records an advance payment given to an employee. HR, ADMIN GET / api /v1/ hr /advances/{ employeeId } Retrieves advance payment history for tracking. Database Design Considerations Supplier_Item_Mapping  Table:  A junction table between Suppliers and Items containing the  supplier_id ,  item_id , and  default_supplier_price . Inventory_Ledger  Table:  Must include a status column (e.g., RAW, WIP_MOLDING, WIP_ASSEMBLE, FINISHED_GOOD) and an  is_damaged   boolean  to satisfy the QC and department tracking requirements. Invoices Table:  Needs an  invoice_type   enum  (TAX, NORMAL), a  tax_amount  column, and a  discount_amount  column to handle the dynamic invoicing logic. Frontend: Here is the structural blueprint for how this is implemented in React. 1. The Core Component Architecture The application should be wrapped in an Authentication Provider that decodes the JWT and stores the user's role in memory. A Higher-Order Component (HOC) or wrapper component (like  ProtectedRoute ) acts as the gatekeeper for the routes. Folder Structure Concept: src / ├ ──  contexts/ │   └──  AuthContext.tsx         // Decodes JWT, provides user  { id , role,  token } ├ ──  components/ │    ├ ──  shared/                // Sidebar, Navbar, Generic Tables │   └── guards/ │       └──  RoleGuard.tsx       // Checks if  user.role  matches allowed roles ├ ──  pages/ │    ├ ──  stock-keeper/ │   │    ├ ──   Dashboard.tsx       // Lazy-loaded entry point │   │    ├ ──   StockInForm.tsx │   │   └──  PoManager.tsx │    ├ ──  molding/ │   │    ├ ──   Dashboard.tsx │   │   └──  WipTracker.tsx │   └── ...other roles └──  App.tsx                     // React Router configuration 2. Implementation Details for the Dashboards Stock Keeper Dashboard:  Needs real-time connection to the  InventoryService . It uses components like < SupplierItemMapper  /> and < MultiItemPoForm  />. When  StockInForm  is submitted, it triggers the API call that eventually fires the  StockInRecorded  event on the backend. Manufacturing Dashboards (Molding, Assemble, Primary):  These are heavily reliant on the  WorkflowService . Their main component is < WipKanbanBoard  /> or a specialized < WipGrid  />. They need a specific UI element (e.g., < AdvanceBatchModal  />) to move items from one state to another. QC Dashboard:  This dashboard queries the inventory but filters specifically for the  isDamaged  flags and pending inspections. It uses < DamageSegregationTool  /> to separate bad batches from good ones before they enter the main live stock.
